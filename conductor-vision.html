产品层 · 大脑 Conductor Work 执行内核 · 引擎 Aeloop

面向企业研发的可信 AI 工作流平台。它解决的不是「再加一个聊天机器人」——而是把 AI 从一次性对话,升级成可控、可审、可恢复的工作流。

冻结需求为任务契约 多模型独立复核 人工闸门 · 跨进程恢复 每步证据与 Token 账本 fail-closed 安全策略

从 Verity 走到今天

同一个初心,换了一副骨架

上一版 Verity(2026-07 提案)是一套骑在 Claude Code 之上的治理层——用 hook、CLAUDE.md、Python 脚本把规则钉在单一模型上。方向对,但大脑和引擎焊死在一起,换模型即重写。这一版把它切成两层。

怎么拆出来的——Verity 把「要做什么」(需求编译、风险判断、产品交互)和「怎么可靠执行」(模型、上下文、闸门、恢复、审计)混在一块。这一版沿这条缝切开:上半段成为 Conductor Work(可替换的产品大脑),下半段成为 Aeloop(模型无关、可开源的执行内核);两者之间用 TaskContract 往下、EvidenceBundle 往上通信。公司版 checkout 可以整块移除个人材料,而不动引擎。

业务问题

障碍不是「AI 不会生成」,是生成结果进不了可控流程

01

需求漂移

模型补充 PRD 没要求的功能,或悄悄遗漏边界条件。

02

幻觉与假验证

声称「已运行测试」,却没有可追溯的证据。

03

上下文断裂

换聊天、换模型或中断后,要从头重新解释背景。

04

缺乏独立复核

同一模型生成又自审,错误模式高度相关,审了等于没审。

05

安全风险

自动工具可能改错文件、引未批准依赖、执行远端 Git 操作。

06

不可复用 · 不可见成本

提示散落临时聊天,无法版本化;Token / 延迟 / 重试无账可查。

防幻觉与可信机制

把「自觉要求」换成可执行机制

模型声明不直接视为事实。每一句「做到了」,都要能落到具体命令、退出码和输出摘要。

◇ Claim-to-Evidence 证据链

Requirement → Claim → Changed Artifact → Tool Execution → Exit Code / Hash → Verification

每条声明都被打上明确状态,不允许「大概过了」这种模糊态:

verified failed not_proven stale

◉ 独立模型复核

生产者 ≠ 审查者,从机制上切断相关错误。

  • Coder 与 Tester 使用不同模型
  • Tester 只读,不继承 Coder 的全部推理上下文
  • 按 Requirement ID 给 PASS / FAIL / NOT_PROVEN
  • 连续失败达阈值升级给人,不无限烧 Token
需求覆盖矩阵 · Requirement Coverage
Requirement Implementation Evidence Result
REQ-001 src/features/search/… test:status-filter · exit 0 PASS
REQ-002 未找到完整实现 无 NOT_PROVEN
安全与合规边界

默认 fail-closed:没明确允许,就是拒绝

Git 操作 永久禁止

  • ✕ git commit
  • ✕ git push
  • ✕ 创建 Pull / Merge Request
  • ✕ merge
  • ✕ 修改远端分支 / 标签

引擎永不自动写远端。未来若要自动化 SCM,必须由独立产品模块 + 新审批政策实现,不进内核。

Workspace 隔离

  • ✓ 每个 Run 独立隔离工作区
  • ✓ 记录 baseline commit 与初始 dirty 状态
  • ✓ 从错误目录恢复 → 拒绝继续(非仅警告)
  • ✓ Reviewer 只读
  • ✓ 产出 = 本地候选变更 + 证据

依赖 Allowlist

  • ✓ 只允许公司批准的包
  • ✓ 只允许批准的内部 registry
  • ✕ 不在清单的依赖 → 直接失败
  • ✓ 依赖策略版本随 Run 固化进审计

网络与数据 Allowlist

  • ✓ 默认无网,或仅 LiteLLM / 内部 GitLab / 批准 Figma
  • ✓ 凭据来自环境或公司 secret manager
  • ✕ PRD / Figma / 仓库内容 / 日志不入公共代码仓
  • ✓ 日志可配置脱敏与保留周期

选型对照

已经有 Ruflo,为什么还要自研 Aeloop

Ruflo(第三方开源 ruvnet/claude-flow 的本地版)是一套「什么都能接」的 Agent 元框架;Aeloop 走的是「一条严格可信流程做透」。两者不是竞品——Ruflo 是借鉴与参考源。

维度 第三方 · 社区维护Ruflo / claude-flow 自研 · 自己掌控Aeloop
来源 第三方开源(ruvnet/claude-flow),数千次提交、社区维护 自研,边界与路线自己定
定位 Agent 元框架:堆能力(海量 MCP 工具 / 近百 agent / swarm / federation) 治理优先的 coder/tester 引擎,治理焊进内核
广度 vs 深度 广:什么都能接,能力多而杂 深:契约→证据→闸门→审计一条链做透
治理 能力导向,治理不是第一性 契约 / 证据 / fail-closed / 审计是第一性
模型 偏 Claude Code / Codex 生态 模型无关(LiteLLM,换 Profile)
维护成本 ≈ 零(社区扛) 自己扛——但换来可控
对 Aeloop 的角色 借鉴 / 参考源,不是竞品 我们要下注的差异化
▸

一句话:Ruflo 赌广度,社区已经在替我们跑;Aeloop 的赌注是「深度 + 治理」——把「AI 干活但每一步可审、可拦、不能靠嘴糊弄」这件事做成确定性能力,这是买不来、也是 Ruflo 那条路线不优先解决的。

为什么先押 Aeloop

两层里,先把引擎钉死

不是不做产品层,而是产品层需要一块稳定地基。先做引擎的四个理由:

01 · 地基先稳

引擎边界钉死,产品层才不会盖在流沙上

先完成公共契约 / 事件 / workspace / evidence / workflow 缝的重构,再用公司 LiteLLM 做真实验收——避免在旧边界上固化 profile 行为,回头推倒重来。

02 · 风险最小

引擎不含凭证与私人材料,可独立开源、独立验收

公司 checkout 只需满足 config.yaml 契约;PRD / 仓库内容 / 记忆库都留在产品层。引擎这块拿出来给人看、给人审都安全。

03 · 打最痛的点

coder-tester 是当前最痛、最可测的场景

先用它把「契约 → 证据 → 闸门 → 审计」全链真实跑通,拿到确定性价值,再谈扩 research / prd / design 等其它 workflow。

04 · 差异化在这

差异化赌注押在引擎,不在能力堆叠

广度那条路 Ruflo / 社区在走;我们的价值在治理深度。先把引擎做出别人抄不走的确定性,再考虑规模与产品化。

已经能证明的是:pnpm run demo:company 跑通了「校验公司契约 + 确定性选中 workflow」——不调模型、不碰仓库、零公司凭证,证明这套两层骨架本身立得住。下一步(A6)才是接 LiteLLM 做双 Profile 真实验收。

价值与差异化

不是「多一个 Agent」,是一种受控的 AI 工作方式

聊天上下文 → 持久任务契约
模型自述 → 证据链
单模型自审 → 独立模型复核
一次性 Prompt → 可版本化 Workflow
人工反复解释 → 跨进程恢复
不可见消耗 → 按任务统计 Token / 时间 / 成本
自由工具调用 → 企业安全策略

最终目标:让 AI 自动化从「看起来能做」变成「过程可控、结果可查、失败可恢复、人仍掌握最终决定」。
