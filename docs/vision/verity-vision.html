问题 02 / 上下文疲惫

重新介绍项目背景、技术栈、上次做到哪。你不是在和搭档合作，你是在不停给一个失忆的临时工做入职培训。

问题 03 / 工具割裂疲惫

每个工具都聪明，但没人管它们怎么接起来

公司的 AI 操作现在分散在一堆独立 skill/tool 里——文档、设计稿转代码、测试用例生成，每个单独都好用。但一个项目做下来，「决定用哪个工具、把上一个工具的产出改成下一个能读的格式、跑完再手动拼结果」本身占用的时间越来越多。工具越多，这层拼接成本涨得越快，且没人管。

机制层

不是纸面规则，是真实在跑的代码

目录结构（真实存在的文件，非规划） 

ai-agent/
├── CLAUDE.md              # session 启动指令
├── VERITY.md            # 人格宪法
├── .claude/settings.json  # Hook 接线配置
├── scripts/
│   ├── inject-context.py  # SessionStart Hook
│   ├── auto-snapshot.py   # SessionEnd Hook
│   └── memory_db.py       # agent.db 读写封装
├── state/
│   ├── active.md / ideas.md / snapshot.md
│   ├── agent.db            # SQLite + FTS5 检索式记忆库
│   └── .snapshot-failed    # 失败留痕标记（存在即代表上次未成功）
├── maps/                  # 每项目一张地图
└── archive/               # decisions / constraints / postmortem
① SessionStart → inject-context.py 对话开始时，状态自动注入上下文

identity / snapshot / active_task 始终全量注入——量小，是"当下必须知道"的状态。用户消息里的关键词额外触发 FTS5 检索，从 decision / postmortem / map / idea 等类型里按需召回，不再无差别读全部 md 文件。

UserPromptSubmit Hook 全量注入 3 类 FTS5 关键词检索 stale 超 14 天提示
② SessionEnd → auto-snapshot.py 对话结束时，读真实 transcript 生成摘要

Hook 拿到的是 transcript_path（指向 .jsonl 会话记录），不是对话内容本身。脚本逐行解析、过滤子代理会话，取最近 20 轮 user/assistant 消息喂给 Claude API 生成摘要，写回 snapshot.md 并同步 agent.db。

transcript_path 解析 跳过 isSidechain 写 snapshot.md + agent.db
③ 失败留痕，而不是静默假绿 摘要生成失败时，下次对话开场就会看到警告

没有 transcript、没有 API key、或调用失败——任何一种失败都会写 state/.snapshot-failed。下次 inject-context.py 注入时会把这条警告一起带上，并在 snapshot 内容里标注「过期状态，非最新对话的真实反映」。AI 生成的摘要本身也带 unconfirmed 标记，直到 开发者 在下次对话里确认或纠正才算数——这是「零幻觉」原则在机制层的落地，不只是写在宪法里的一句话。

.snapshot-failed unconfirmed 标记 人工确认才算数

记忆库设计理念 — agent.db 为什么这么设计

AI 的检索方式是语义/关键词匹配，不是走房间路径——因此选择一个 SQLite 文件 + FTS5 全文索引，配合两个 Hook 分管注入和写回。

① 分层而非平铺 不是所有记忆一个权重

identity / snapshot / active_task 是「当下必须知道」的状态，量小，无条件全量注入。decision / postmortem / map / idea 等体量会持续增长，靠用户消息里的关键词触发 FTS5 检索才召回——上下文消耗不随记忆库变大而线性膨胀。

② 新鲜度是记忆的一部分 记忆不只有「有没有」，还有「还能信多少」

召回的记忆若超过 STALE_DAYS = 14 天未更新，会带着「⚠️ 该记忆已 N 天未更新，内容可能已过期」一起注入，而不是原样呈现让人误以为是最新状态。

③ AI 写的记忆是草稿，不是事实 unconfirmed 标记 + 人工确认循环

每条自动生成的 snapshot 都打 unconfirmed 标记，下次对话开场必须经 开发者 过一遍才算数。理由：自动摘要一旦被当事实无差别注入，错误会自我强化——一步错，步步错。

与常见做法的区别

常见做法 Verity 的方案
每次对话全量重读所有 md 文件，随文件增多线性变慢 3 类核心状态全量注入，其余按关键词检索，不随历史堆积拖慢单次对话
用向量 embedding 做语义检索 SQLite FTS5 关键词全文检索——本地、零外部依赖、零 embedding 成本
AI 生成的总结直接当事实覆盖旧状态 强制 unconfirmed 标记 + 人工确认循环，防止错误摘要自我强化
静默失败，下次直接用旧状态，没人知道 .snapshot-failed 显式留痕，下次开场白连同警告一起注入
md 文件既是存储也是检索源，两者耦合 md 是人读的镜像，agent.db 是机器读的真相源，解耦互不阻塞

核心区别一句话：这不是「给 AI 一个更大的记忆容器」，而是「承认 AI 的记忆会错、会过期、会漂移，然后把这三件事都做成可见的机制，而不是假装它们不存在」。
