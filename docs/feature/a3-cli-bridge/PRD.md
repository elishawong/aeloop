## 0. Decisions already made (decided by the commander on 2026-07-20, locked in, no further discussion)

1. **v1 `ToolExecVerifier` = existence / subset matching** — "the claimed tool executions ⊆ the real tool calls that actually appear in the trace." **Parameter-level matching (checking whether the specific file paths/command arguments claimed match verbatim against the real calls in the trace) is explicitly pushed to v2**, out of scope for this increment (§2 Non-Goals).
2. **The `permission_denials` field is marked `[?]`** — the spike was unable to reproduce its trigger condition within 2-3 attempts (see spike-findings.md §2.3); v1 does not consume this field, and no adapter is guaranteed to populate/depend on it.
3. **Claude's "no tool call" negative-control case was not run independently in the spike** — the PRD's test plan **must include**: `ClaudeCliAdapter` tested with a prompt that clearly needs no tools → assert the trace is an empty array (symmetric with the negative control already validated for codex), to be added in the first batch of the build phase (§5 test subsection).

## 1. Problem / Users / Approach

- **Problem to solve**: A2 built the Harness layer, but `harness/config.ts`'s `buildAdapterRegistry()` explicitly skips `kind === "cli-bridge"` (the comment literally says: "A3 fills in the construction branch for ClaudeCliAdapter/CodexCliAdapter here") — meaning that for the subscription profile (subscription-based, no API key, both providers `claude-cli`/`codex-cli` are `cli-bridge`), calling `buildAdapterRegistry(subscriptionConfig)` currently yields an **empty** `AdapterRegistry`, with no ability to actually call a model. At the same time, DESIGN §8 explicitly names `ToolExecVerifier` as "the one gate that actually guards against hallucination" — `ClaimSchema.verifiedBy` allows the model to self-report that a given claim was verified via `"tool_execution"`, but currently there's no mechanism to verify whether that self-report is true. A model could brazenly write `verifiedBy: "tool_execution"` having never invoked any tool at all.
- **Who it's for**: The direct consumer is A4 (the Coder/Tester nodes orchestrated by the Loop/LangGraph) — A4 will call `ProviderRouter.route("coder")`/`route("tester")` against the subscription profile to obtain a real `ClaudeCliAdapter`/`CodexCliAdapter` instance and call `invoke()`. Further downstream is Elisha's own dogfooding loop (subscription-based, not billed against the company's API budget). In the short term the direct users are Cypher/Zorro running tests in this increment and in A4.
- **One-line approach**: Build `ClaudeCliAdapter`/`CodexCliAdapter` (both `implements ModelAdapter`, `kind: "cli-bridge"`, sharing a newly built generic subprocess execution primitive `cli-exec.ts` for spawn/timeout/stdin-closing) according to the real CLI invocation shapes validated by the spike, each with its own JSONL trace parser that produces a uniformly shaped `ToolCallRecord[]`; build `ToolExecVerifier` (a pure function `checkToolExecution`) that verifies, per the already-decided "existence matching" rule, whether "the claim asserts tool_execution" is consistent with "a tool call actually happened in the trace," producing `InvokeResult.toolExecChecked`; wire up the cli-bridge branch in `harness/config.ts` so that `profiles/subscription/config.yaml` can actually construct the two real adapters; and finally prove with one hard vertical-slice test that this pathway is genuinely connected end to end (`PromptComposer → ProviderRouter → AdapterRegistry (a real cli-bridge adapter, with the subprocess pointed at a controlled fixture script rather than the real claude/codex binary) → SchemaValidator → ToolExecVerifier's verdict landing in InvokeResult`).

## 2. Goals / Non-Goals

**Goals**:
- `src/harness/types.ts`: replace the `ToolCallRecord` placeholder left by A2 (`{ [key: string]: unknown }`) with its real shape, add a new `ToolExecChecked` type alias (`"pass" | "fail" | "na"`, reused for `InvokeResult.toolExecChecked`).
- `src/harness/cli-exec.ts`: a CLI-agnostic generic subprocess execution primitive — `spawnWithTimeout()`, wall-clock timeout + `SIGKILL` + **immediate `stdin.end()`** (spike §1.6 independently reproduced codex's stdin-blocking pitfall; claude may not have the same pitfall, but guarding against it defensively costs nothing) + stdout/stderr collected **separately** (spike §1.5: in codex's `--json` mode, noise goes to stderr — merging would pollute JSONL parsing; `codex-client.mjs`'s merged handling **cannot be copied as-is**).
- `src/harness/adapters/codex-cli-adapter.ts`: `CodexCliAdapter` — `codex exec --json --sandbox read-only "<prompt>"`, parses `item.completed` events with `type==="command_execution"` into the trace, takes the **last** `agent_message`'s `text` as `content`.
- `src/harness/adapters/claude-cli-adapter.ts`: `ClaudeCliAdapter` — `claude -p "<prompt>" --output-format stream-json --verbose --permission-mode bypassPermissions --allowedTools "<read-only tool set>"`, parses `tool_use`/`tool_result` events into the trace, takes the last event with `type==="result"`'s `.result` field as `content`.
- `src/harness/tool-exec-verifier.ts`: `checkToolExecution(content, trace): ToolExecChecked` — a pure function implementing the already-decided existence-matching rule (precisely defined in §5).
- `src/harness/config.ts`: fill in the `kind === "cli-bridge"` branch — dispatch to construct the corresponding adapter based on `providerConfig.cmd` (`"claude"` | `"codex"`); an unrecognized `cmd` value throws `InvalidProviderConfigError` (consistent with the error-handling style already used in the direct-api branch).
- **Hard vertical slice**: one end-to-end test proving that the chain "a real prompt produced by `PromptComposer` → a real `buildAdapterRegistry` (using a `ProfileConfig` that points at a **controlled fixture script** rather than the real `claude`/`codex` binary) → `ProviderRouter` selects the correct real cli-bridge adapter → the adapter really `spawn`s a subprocess and really parses its JSONL output → `SchemaValidator` validates it successfully → `ToolExecVerifier`'s verdict correctly lands in the final `InvokeResult.toolExecChecked`" is fully connected — not a handful of isolated green tests.

**Non-Goals (explicitly not doing, left to a future increment or v2)**:
- ❌ **Parameter-level matching** (checking whether the specific file paths/command arguments claimed match verbatim against the real calls in the trace) — already decided in §0-1, pushed to v2.
- ❌ **Consuming the `permission_denials` field** — already decided in §0-2, the trigger condition is unverified, v1 does not depend on it.
- ❌ **Porting `codex-client.mjs`'s "untrusted binary path" security invariant** (that file's security invariant ⑤, which rejects execution of a `codex` binary that resolves to the repo itself or a system temp directory) — that is a hardening measure specific to a particular high-risk trust boundary in Zorro's independent-review evidence chain (guarding against someone forging `codex` output to fool the independent review). aeloop's own coder/tester loop has no directly corresponding equal-risk scenario. v1 only does PATH resolution to obtain the absolute path for auditing, and does not implement the "reject execution" layer. **If the strategist/commander believes `ToolExecVerifier`'s trustworthiness itself depends on "the `codex`/`claude` on PATH really being the genuine binary" and therefore needs the same hardening, please say so explicitly during the confirmation phase (§9.1)** — I am not quietly dropping this layer of security on the commander's behalf; I'm judging that the threat models of the two scenarios differ, and flagging it for a ruling.
- ❌ **Specifying the underlying model via `config.yaml`** (a `-m`/`--model` override) — v1 uses whatever default model the CLI itself is already configured with (the spike's actual measurements were `gpt-5.6-sol` (codex)/`claude-sonnet-5` (claude); this is determined by the current local configuration, not hardcoded by the adapter).
- ❌ **Letting the coder role actually write file changes to disk via the CLI** — `CoderOutput.diff` is a **string** artifact (DESIGN §3 sequence: `Coder-->>Orc: {diff, claims[], confidence}`), not "the CLI itself writing the change into the workspace." Both adapters default to a read-only posture (codex `--sandbox read-only`; claude restricts `--allowedTools` to a read-only tool set, precise list in §5). Whether the coder role needs elevated permission to directly edit files is a matter for A4's Loop gateway design, not for A3's adapter defaults.
- ❌ **The Loop layer's G1-G3 gates / persisting `structured_claims`/`workflow_runs`/`approvals` tables** — that's A4; A3 only guarantees that `InvokeResult.toolExecChecked` is computed correctly, and does not touch the database (consistent with A2 §4's "stateless, no tables" boundary).
- ❌ **`profiles/apikey/`** — both roles in the apikey profile go through `LiteLLMAdapter` (direct-api); A3 doesn't touch it.
- ❌ **Real CLIs entering the automated test suite** — per `CLAUDE.md`'s "remote ignition" principle (programmatic/automated calls must not hit an interactive, subscription-billed CLI). All tests go through **controlled fixture subprocess scripts** (full design in §5/§6); no test actually `spawn`s the production `claude`/`codex` binary.

## 3. User Stories

- As an **A4 Loop developer**, I want calling `ProviderRouter.route("coder")` against the subscription profile to directly hand me back a `ModelAdapter` that can genuinely invoke the `claude` CLI, without having to worry about spawn/JSONL-parsing details.
- As an **A4 Loop developer**, I want `InvokeResult.toolExecChecked` to clearly return `"fail"` when the model falsely claims "I verified it" but actually invoked no tools at all, rather than silently trusting the model's self-report.
- As the **commander**, I want to see a test proving that a hallucination scenario — "the model self-reports `verifiedBy: 'tool_execution'` but the trace is completely empty" — genuinely gets caught and marked `fail` by `ToolExecVerifier`. Not a claim in the docs, but an actual test run proving it.
- As the **commander**, I want confidence that this set of adapters won't secretly hit the real `claude`/`codex` and burn subscription quota inside the automated tests — the test strategy needs to spell this out explicitly.

## 4. Data Model

This increment is **stateless, no tables**, consistent with A2 — `structured_claims`/`workflow_runs`/`approvals` remain A4 Loop's concern. The only "data shape" change is at the in-memory type level:
- `src/harness/types.ts`'s `ToolCallRecord` (placeholder replaced with a real shape, defined in §5).
- `InvokeResult.toolExecChecked` — a field already declared by A2 but never populated — is populated for the first time in this increment by a real adapter, as `"pass" | "fail" | "na"` (A2's direct-api adapter still does not set this field, and continues to leave it `undefined` — a distinction already made clear in DESIGN/types.ts: `undefined` means "this adapter has no verification capability," an explicit `"na"` means "this adapter has verification capability, but there was nothing to verify this time." A cli-bridge adapter belongs to the latter category and must not use `undefined` to paper over that).

## 5. File-by-File Task List

### Types / generic primitives

- `src/harness/types.ts` (**modify**, not a new file):
  - Replace the placeholder `ToolCallRecord` definition left by A2. Real shape (evidence from both CLIs both fit, spike-findings.md §3.1 comparison table):
    ```typescript
    export interface ToolCallRecord {
      /** Unified tool identifier. codex always fills in "shell" (codex --json only exposes
       * shell-level command_execution and can't distinguish which specific underlying tool
       * was called — spike-findings.md §3.1); claude fills in the real tool name (Bash/Read/...). */
      toolName: string;
      /** The order in which this call appeared in the event stream collected during this
       * invoke() call (0-based) — the basis for establishing the premise "happened before the
       * final content" (the core finding of spike-findings.md §1.4). */
      sequenceIndex: number;
      /** Whether this call itself succeeded: codex uses exit_code===0; claude uses
       * tool_result.is_error===false. Left undefined when the underlying CLI gives no clear
       * success/failure signal — don't guess. */
      succeeded?: boolean;
      /** The raw event object, for debugging/auditing; ToolExecVerifier itself does not parse
       * this field any further. */
      raw: Record<string, unknown>;
    }
    export type ToolExecChecked = "pass" | "fail" | "na";
    ```
  - Change `InvokeResult.toolExecChecked`'s type to use `ToolExecChecked` (avoids defining the string-literal union twice in two places).
- `src/harness/cli-exec.ts` (**new file**):
  - `spawnWithTimeout(cmd: string, args: string[], opts): Promise<{ exitCode: number | null; signal: string | null; stdout: string; stderr: string; timedOut: boolean }>` — a CLI-agnostic generic subprocess execution primitive, shared by both adapters.
  - Mirror the parts of `scripts/openai/codex-client.mjs` already validated: an explicit wall-clock `setTimeout` + `SIGKILL` on expiry; **immediately call `child.stdin.end()`** (spike §1.6 independently reproduced codex's stdin-blocking pitfall in plain command-line testing — this is not a theoretical risk); stdout/stderr collected **separately**, not merged (spike §1.5, and **different** from how `codex-client.mjs` handles it — cannot be copied as-is); an output byte-size cap (mirroring `codex-client.mjs`'s 32MB cap, to guard against memory blowup).
  - **Not mirrored**: `codex-client.mjs`'s git-HEAD / file-integrity snapshot checks, and `resolveCodexBinary`'s "untrusted path" execution-rejection logic — those belong to a trust boundary specific to the review-wrapper (see §2 Non-Goals). `spawnWithTimeout` only does one thing: "run a command, reliably get its output, never hang."
  - `opts.spawnImpl` (default `node:child_process.spawn`) can be injected, for this file's own unit tests to use (§6 test subsection); **not exposed externally to either adapter's public constructor** — adapter-level tests go through real subprocesses (see the overall "test strategy" section below); there's no need to open another injection point at the adapter layer.

### `ToolExecVerifier`

- `src/harness/tool-exec-verifier.ts` (**new file**):
  - `checkToolExecution(content: string, trace: readonly ToolCallRecord[]): ToolExecChecked` — a pure function; the concrete v1 landing of the already-decided rule (§0-1):
    1. Attempt `JSON.parse(content)`; if parsing fails, or the parsed object has no `claims` array → return `"na"` (**not** `"fail"` — this situation is generally the kind of case `SchemaValidator` would retry/fall back on; `ToolExecVerifier` shouldn't guess at a negative verdict when it "doesn't understand" the input).
    2. If **any** entry in the `claims` array has `verifiedBy === "tool_execution"` → treat this as "this response claims a tool_execution happened"; otherwise → return `"na"` (nobody claimed it, nothing to verify).
    3. If it was claimed → `trace.length > 0` ? `"pass"` : `"fail"`.
  - **This is the honest landing of "existence matching" given the current `ClaimSchema` shape, not an arbitrary simplification**: `ClaimSchema` (`src/prompt/schema.ts`) currently only has the boolean-like self-reported field `verifiedBy: "tool_execution" | "human" | "unverified"`, and **does not** have a field for "which specific tools were claimed to have been called" (such as `toolsUsed: string[]`). So the literal wording of decision §0-1 — "the claimed tool types ⊆ the tool types that actually appeared in the trace" — can, under the current schema, only be implemented in one honest form: "claimed to have happened ⊆ actually happened" (a subset check on a single-element set, degenerating into an existence check). **I believe this is the only honest implementation of the decision's intent under the current schema, but this is my interpretation, not something the decision text literally required — please confirm during the confirmation phase whether this matches expectations.** If the commander/strategist wants v1 to add a `toolsUsed?: string[]` field to `ClaimSchema` for finer-grained matching, that's a change to A1's schema (out of scope for A3's file list, and would require updating two persona documents) — that would need to be separately decided as in-scope or not for this increment.
  - **"Temporal precedence" is automatically satisfied under this design and needs no extra code**: the pitfall discovered in spike-findings.md §1.4 (codex mid-stream self-reporting `tools_used:[]`, only correcting itself once it actually finished) is fundamentally a matter of "content extraction must grab the last one, not an arbitrary one" — as long as `CodexCliAdapter`/`ClaudeCliAdapter`'s own content-extraction logic strictly takes "the very last final answer in the entire event stream" (see each adapter's task description below), then the `trace` passed into `checkToolExecution` is naturally "all tool calls that happened during this entire invoke() call, all of which precede that final content" (in a non-interactive single-turn call, the event stream is strictly time-ordered — `turn.completed`/`type:"result"` is always the last line) — `checkToolExecution` itself does not need to do any timestamp comparison.
- `src/harness/__tests__/tool-exec-verifier.test.ts`: covers four combinations — ① claims tool_execution and trace is non-empty → `"pass"`; ② claims it but trace is empty → `"fail"` (**this is the single most important test in this PRD** — it directly corresponds to the anti-hallucination scenario "the model falsely claims it verified something but never invoked any tool at all"); ③ no claim asserts tool_execution → `"na"`; ④ `content` is not valid JSON / has no `claims` field → `"na"`, no exception thrown.

### `CodexCliAdapter`

- `src/harness/adapters/codex-cli-adapter.ts` (**new file**):
  - Constructor: `new CodexCliAdapter(id: string, config: { cmd?: string })` (`cmd` defaults to `"codex"`, taken from `ProviderConfig.cmd`, already `${ENV}`-substituted). `kind = "cli-bridge"`.
  - `invoke(req)`: calls `spawnWithTimeout(config.cmd, ["exec", "--json", "--sandbox", "read-only", req.prompt], { timeoutMs: DEFAULT_TIMEOUT_MS })` (default timeout constant mirrors `codex-client.mjs`'s 600s, discussed in §9.2; **does not add** `--skip-git-repo-check` — production always runs inside a real repo, that flag is only needed because this spike's test directory wasn't a git repo).
  - Parse stdout (**must be the separately-collected copy, without stderr noise** — spike §1.5): line-by-line `JSON.parse`, skipping lines that fail to parse (tolerance, not every line is guaranteed to be valid JSON — e.g. there's a tiny chance a codex banner/warning leaks into stdout);
    - trace: collect all events with `type==="item.completed"` and `item.type==="command_execution"`, mapped in order of appearance into `ToolCallRecord` (`toolName: "shell"`, `sequenceIndex` = order of appearance, `succeeded: item.exit_code === 0`, `raw: item`).
    - content: take the **last** event with `type==="item.completed"` and `item.type==="agent_message"`'s `item.text`. If none is found → treat as abnormal output, throw `AdapterInvokeError` (don't return an empty string to paper over it).
    - `model`: **`[?]` new finding (beyond the original scope of spike-findings.md, re-verified during PRD writing) — the JSONL event stream from `codex exec --json` contains no `model` field at all** (re-verified across all `--json` samples captured during the spike using `grep -o '"model"'`, zero matches; the line `model: gpt-5.6-sol` only appears in the human-readable banner **without** `--json` — that banner section isn't printed in `--json` mode). **v1 approach**: hardcode `InvokeResult.model` to `"unknown"` (mirroring `codex-client.mjs`'s own existing convention in `buildAttestation` of `model: model ?? 'unknown'`, satisfying `types.ts`'s hard constraint that "provider/model must be a non-empty string" while not fabricating a fake model name). If some future codex version's `--json` output starts including a model field, this can be revisited.
  - `provider` is always filled with `this.id` (the same convention as `LiteLLMAdapter`).
  - `toolExecChecked`: `invoke()` internally calls `checkToolExecution(content, trace)` directly to populate it.
  - `checkAvailability()`: `spawnWithTimeout(config.cmd, ["--version"], { timeoutMs: <a short timeout, e.g. 10s> })`, exit code `0` → `available: true`; non-zero / spawn failure (ENOENT) / timeout → `available: false` + reason (not just checking whether `config` has `cmd` filled in and calling it done — this directly corresponds to the lesson from DESIGN §8.5 "visible in the deepseek list ≠ actually callable," and is consistent with `LiteLLMAdapter.checkAvailability()`'s existing convention).
  - `toolTrace()`: returns the `ToolCallRecord[]` collected by the **most recent** `invoke()` call (a private internal field stores the most recent result; `invoke()` resets it to `[]` at the start, then populates it as parsing proceeds; returns `[]` if `invoke()` has never been called, never throws). **Known limitation, documented in code comments**: this state is shared per instance — if the same adapter instance has `invoke()` called concurrently multiple times, `toolTrace()` will have a race condition (which call's result gets read is undefined). v1 does not handle the concurrent case — DESIGN's coder→tester loop is itself sequential, and A4 Loop's current calling pattern has no scenario of "the same adapter instance being invoked concurrently twice"; if this arises in the future, the trace would need to move onto `InvokeResult` itself rather than being retrieved via a query-style `toolTrace()`, but that's an interface-level change out of scope for A3 to solve — this is just an honest record of the limitation.

### `ClaudeCliAdapter`

- `src/harness/adapters/claude-cli-adapter.ts` (**new file**):
  - Constructor: `new ClaudeCliAdapter(id: string, config: { cmd?: string })` (`cmd` defaults to `"claude"`). `kind = "cli-bridge"`.
  - `invoke(req)`: calls `spawnWithTimeout(config.cmd, ["-p", req.prompt, "--output-format", "stream-json", "--verbose", "--permission-mode", "bypassPermissions", "--allowedTools", "Bash,Read,Grep,Glob"], { timeoutMs: DEFAULT_TIMEOUT_MS })`.
    - `--verbose` is strictly mandatory (spike §2.4 confirmed: `--output-format stream-json` without `--verbose` fails immediately with an error — not optional).
    - `--permission-mode bypassPermissions` is strictly mandatory (spike §2.4: in non-interactive mode, without explicitly declaring this, behavior depends on whatever permission state the calling environment already has — not portable, and may hang waiting for approval).
    - `--allowedTools "Bash,Read,Grep,Glob"`: **the read-only tool set** (corresponding to codex's `--sandbox read-only` posture, rationale explained in §2 Non-Goals). **Honestly noting one limitation**: this is not an OS-level sandbox — the `Bash` tool itself could in theory still run destructive commands like `rm`. The claude CLI's permission system operates at the level of "can this tool be invoked at all," not the kind of process-level read-only mount that codex's `--sandbox` provides. This difference is documented in code comments; whether stronger isolation is needed (e.g. running the entire adapter inside a dedicated sandboxed environment) is `[?]` marked and left to A4/the strategist to evaluate — not something A3 can unilaterally resolve.
  - Parse stdout (line-by-line `JSON.parse`, skipping lines that fail to parse):
    - trace: collect all `tool_use` entries found in `message.content[]` of `type==="assistant"` messages, and pair them with the corresponding `tool_result` entries found in `message.content[]` of subsequent `type==="user"` messages via `tool_use_id`/`id`, mapped into `ToolCallRecord` (`toolName: tool_use.name`, `sequenceIndex` = order of appearance of the `tool_use`, `succeeded: tool_result.is_error === false` (left `undefined` when no matching `tool_result` is found), `raw`: both events stored).
    - content: take the **last line** with `type==="result"`'s `.result` field. Check `.subtype==="success"` and `!is_error`; otherwise throw `AdapterInvokeError` (including whatever error info can be gleaned from `.result`/`.subtype`).
    - `model`: take the `.model` field from the `type==="system"`, `subtype==="init"` event (spike measurements show this event is always the true first content event of the stream, present even for a pure-text response with no tool calls — more reliable than digging it out of some particular assistant message). If not found → `"unknown"` (same fallback strategy as the codex adapter, no fabrication).
  - `provider`/`toolExecChecked`/`checkAvailability()`/`toolTrace()`: same design as the corresponding parts of `CodexCliAdapter` (`checkAvailability()` uses `<cmd> --version`, judged by exit 0; `toolTrace()` has the same "most recent call only, not concurrency-safe" limitation).

### `harness/config.ts` (**modify**)

- Change the `case "cli-bridge":` branch from "explicitly skipped" to actually constructing:
  ```typescript
  case "cli-bridge": {
    const cmd = providerConfig.cmd;
    if (cmd === "claude") { registry.register(new ClaudeCliAdapter(id, { cmd })); break; }
    if (cmd === "codex") { registry.register(new CodexCliAdapter(id, { cmd })); break; }
    throw new InvalidProviderConfigError(
      id, `cli-bridge provider "cmd" must be "claude" or "codex", got ${JSON.stringify(cmd)}`,
    );
  }
  ```

### Tests (mapped one-to-one to the file-by-file task list)

**Test strategy (spelled out up front — three layers, each with its own boundary, not arbitrarily chosen)**:
1. `cli-exec.ts`'s own unit tests — inject `spawnImpl` (an EventEmitter fake subprocess, mirroring `codex-client.test.mjs`'s `mockSpawn` technique), specifically testing "timeout → SIGKILL," "immediate stdin.end()," "stdout/stderr collected separately" — **generic mechanisms that have nothing to do with a real subprocess** — no need to actually wait for a real process to time out (use a small `timeoutMs: 50` + a fake subprocess that never emits `close`, the same technique already validated in `codex-client.test.mjs`).
2. Unit tests for both adapters — **do not** inject `spawnImpl`; instead point `cmd` at a **controlled fixture script** (`src/harness/adapters/__tests__/fixtures/fake-codex.fixture.mjs` / `fake-claude.fixture.mjs`, plain Node `.mjs` scripts that don't participate in `tsc` compilation, reading an environment variable to choose which preset JSONL output to print before exiting). This way `spawnWithTimeout` **really does** `child_process.spawn` a **real** subprocess — it's just that this subprocess is a stand-in we wrote ourselves: controllable, zero-cost, no network. This mirrors the same "real but controlled" philosophy as A2's `LiteLLMAdapter` tests, which spin up a real local `node:http` server rather than mocking `fetch` — just with the boundary shifted from "network" to "subprocess." The JSONL printed by the fixture scripts is **copied verbatim from the real captured samples in spike-findings.md §1.3/§1.4/§2.2**, not casually made up.
3. Vertical-slice e2e — also goes through fixture scripts (not real binaries), but apart from that single substitution, everything else is real components (see the vertical-slice batch description in §6).
   > **Why not test against the real `claude`/`codex` binaries**: `CLAUDE.md`'s "remote ignition" rule explicitly forbids programmatic/automated invocation of interactive, subscription-billed CLIs (it would cause unintended quota consumption, and be non-deterministic — real model responses can differ each time, which would make tests flaky). The fixture-script strategy lets the test suite **exercise the real subprocess spawn/parse code paths** while staying deterministic, zero-cost, and CI-friendly. This is a deliberate testing-strategy choice made in this PRD, not a shortcut taken out of laziness.

- `src/harness/__tests__/tool-exec-verifier.test.ts`: see the "ToolExecVerifier" section above, all four combinations covered.

## 9. Dependencies / Risks

### 9.1 Whether to port `codex-client.mjs`'s "untrusted binary path" security check — recommend the commander/strategist rule on this

`codex-client.mjs`'s security invariant ⑤ (resolveCodexBinary + rejecting execution of a binary resolving to the repo itself or a system temp directory) exists to prevent someone from forging a fake `codex` to fool Zorro's independent-review evidence chain — that is a **one-shot, high-risk trust boundary whose outcome directly determines "whether the code can be merged."** A3's `ClaudeCliAdapter`/`CodexCliAdapter` serve aeloop's own coder/tester loop — in theory, if the `codex`/`claude` on PATH were maliciously replaced, `ToolExecVerifier`'s verdict would also be poisoned (a forged binary could fabricate "I called a tool" fake events in the trace, causing `checkToolExecution` to mis-judge it as `"pass"`) — this is the same class of risk as the Zorro scenario, just with a different trigger frequency/blast radius (the coder/tester loop's output still passes through the G1-G3 human-approval gates, so a forged `toolExecChecked: "pass"` isn't the sole line of defense; whereas Zorro's independent review is frequently the key judgment call of "can this code be trusted directly"). **My judgment**: don't do this in v1 (YAGNI + keeping A3's own complexity down), but explicitly state that this risk is not "nonexistent," it's "assessed and deferred" — if the commander believes `ToolExecVerifier`'s trustworthiness itself warrants this layer of hardening, please directly amend this section's conclusion; I will not quietly add or quietly skip it in the implementation that follows.

### 9.2 Default timeout value — `[?]` pending confirmation, recommend following `codex-client.mjs`'s existing convention

`cli-exec.ts`'s `DEFAULT_TIMEOUT_MS` is recommended to directly follow `codex-client.mjs`'s 600s (review prompts are typically larger than research prompts; aeloop's coder/tester prompts may similarly come with full context injection, of a comparable magnitude). This value is currently not required to be precisely any particular number by any A3 acceptance item; whether it should become a knob overridable via `config.yaml` (rather than a hardcoded constant) is left for v2 — v1 hardcodes it, YAGNI, there's no known scenario requiring different timeout lengths per role/provider.

### 9.3 codex `--json` has no `model` field — a new discovery made during PRD writing, countermeasure already recorded in §5

This point was not within the original scope of `spike-findings.md` (the original spike only verified that plain-text mode can extract `model` via regex from the banner, and didn't specifically check whether `--json` mode also has this field). The PRD-writing phase re-did a grep review of the already-captured `--json` samples, confirming zero matches. The countermeasure has already been decided in §5's `CodexCliAdapter` task description (hardcode to `"unknown"`); this section just calls this "new finding" out separately for the strategist to review — if deemed useful, this point could be appended to `spike-findings.md`'s open items (§5) to keep the evidence document complete (this PRD does not currently go back and edit that document — left to the strategist's judgment).

### 9.4 `ClaimSchema` currently does not support "which specific tools were claimed" — affects how precisely §0-1's decision can be implemented

See the detailed explanation in §5's `tool-exec-verifier.ts` task description: v1's "existence matching" effectively degenerates into the boolean judgment "claimed tool_execution ⊆ tool_execution actually happened," rather than the true set-subset judgment of "the claimed specific tool list ⊆ the specific tool list in the trace" — because `ClaimSchema` itself carries no "specific list of tool names" field. This is not something I can decide on A1's behalf (`ClaimSchema` is within A1's file scope) — it's flagged here only to confirm whether this understanding matches the intent of decision §0-1.
