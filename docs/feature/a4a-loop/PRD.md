## 0. Scope split: A4a / A4b (direction already set by the strategist + commander, this section documents the rationale for the split)

**The split already decided** (from the dispatch instructions, not my own call):
- **A4a** (this PRD): graph + coder/tester nodes + G1/G2/G3 gates (interrupt/resume, wired via the real checkpoint mechanism proven by the spike) + happy-path vertical slice.
- **A4b** (next PRD round): the threshold-escalation hard branch + persisting the three audit tables (`workflow_runs`/`structured_claims`/`approvals`) to disk + productionizing cross-process checkpoint resume.

**Further subdivision I made within this split** (this is my own judgment call, not directly given by the dispatch instructions ‚Äî flagging it for confirmation): besides "threshold escalation," DESIGN ¬ß4's full state machine has a few more branches that only make sense once the Escalation node exists ‚Äî I've grouped those into A4b too, with rationale in the table below:

| DESIGN ¬ß4 element | Decision | Rationale |
|---|---|---|
| `Draft‚ÜíG1‚ÜíReview‚ÜíG3(approved)‚ÜíApply‚ÜíEnd` | **A4a does this** | The main happy path, explicitly required by the dispatch instructions |
| `G1 reject‚Üífeedback back to Draft` | **A4a does this** | One of the G1 gate's two branches; without it, it isn't really "a gate" |
| `Review sends back‚Üíreject_count+=1‚ÜíG2` | **A4a does this (the count itself)** | The `Inc` step doesn't depend on the Escalation node ‚Äî it happens unconditionally whenever Review sends something back, which is a separate concern from the threshold check (the `T` decision) and can be cleanly split off |
| `count>=threshold hard branch` | **A4b** | Explicitly assigned to A4b in the original dispatch instructions |
| `G2 approve‚ÜíFix(=Draft)‚ÜíReview` | **A4a does this** | The G2 gate's normal branch aside from the threshold |
| `G2 proactive escalation‚ÜíEsc` | **A4b (A4a leaves only the interface, not wired up)** | This branch's destination is the Escalation node; A4a does not build that node. If an external caller passes this decision value in A4a, it explicitly throws `UnhandledGateDecisionError` (fail loud) ‚Äî it is not silently swallowed or silently wired to a branch that doesn't yet exist |
| `G3 reject‚Üínew issue‚ÜíDraft` | **A4a does this** | One of the G3 gate's two branches |
| `Escalation` node + `HD` human decision + `Cancel` terminal state | **A4b** | All three only serve "we've entered escalation" ‚Äî A4a never reaches this point, so building it early would mean building dead code that can't be exercised by tests |
| `Apply` ("write files to the workspace, no auto git commit/push") | **A4a does this, but scaled down** (see ¬ß2 Non-goals) | Actually persisting file changes to disk is an independent, fairly complex diff/patch-application problem, a different concern from "the LangGraph orchestration itself" ‚Äî A4a's Apply node only terminates state, it does not touch the filesystem |

**Conclusion**: what A4a delivers is **everything in DESIGN ¬ß4's state machine except the Escalation subtree**, including the normal branches of all three gates (G1/G2/G3) and the complete round trip between Draft‚ÜîReview (not just "the path that's never been sent back once"). The vertical slice (the e2e test) itself only runs the happy path (passes once, never triggers a send-back), but **the graph structure and the G2/reject branches must genuinely exist and have their own test coverage** ‚Äî this is the difference between "a happy-path vertical slice" and "a graph that only supports the happy path," and the latter is not this PRD's intent.

## 1. Problem / Users / Solution

- **Problem to solve**: A0-A3 have already built and vertical-slice-verified the three-layer `Prompt ‚äÇ Context ‚äÇ Harness` stack, but **nothing yet actually drives the loop of "coder writes a draft ‚Üí human review ‚Üí tester reviews ‚Üí human sign-off"** ‚Äî `ProviderRouter.route("coder")`/`.route("tester")` can get a real adapter, `SchemaValidator` can validate a single call's output, but the orchestration logic for "should this go to the tester after coder finishes," "if tester sends it back, should coder try again," "at what point does this need to go to a human" ‚Äî none of that exists at all. The gate-control state machine DESIGN ¬ß4 draws (Draft‚ÜíG1‚ÜíReview‚ÜíG2/G3) is currently just a diagram, not code. Meanwhile the spike has already proven that LangGraph's `interrupt()`/`Command({resume})`/`SqliteSaver` mechanism works on this repo's stack, but **it only verified the simplest single-gate scenario** (one G1, no conditional edges, no loop-back edges, no G2/G3) ‚Äî `addConditionalEdges` (one of the core mechanisms of DESIGN ¬ß4's state machine) was never touched by the spike at all.
- **Who it's for**: The direct consumers are this increment's own vertical-slice tests and the subsequent A4b (which will add threshold branching + audit persistence on top of the graph A4a builds). Further downstream is A5 (the CLI/TUI layer, which will hook a real human-usable approval interface onto this graph) and, ultimately, Elisha's own actual dogfooding loop.
- **One-line solution**: Build `src/loop/` per the API shapes the spike proved out ‚Äî `nodes/coder.ts`/`nodes/tester.ts` reuse A2's `ProviderRouter`/A1's `PromptComposer`/A2's `SchemaValidator` (no reinventing model invocation); `gates.ts` implements the G1/G2/G3 gates using the `interrupt()`/`Command({resume})` pattern proven in spike Q3, where each gate can genuinely pause and genuinely be resumed by a fresh function call from outside (not depending on the caller retaining any closure-held state); `graph.ts` uses `addConditionalEdges` (unverified by the spike ‚Äî A4a's very first batch must zero out this risk) to wire the six nodes Draft/G1/Review/G2/G3/Apply into a complete state machine per DESIGN ¬ß4 (minus the Escalation subtree); `checkpoint.ts` wires up the `SqliteSaver.fromConnString(path)` pattern proven in spike Q4; finally, one hard vertical-slice test proves the entire chain end to end ‚Äî "a real prompt assembled from real Context‚ÜíPrompt ‚Üí a real cli-bridge adapter built by real `buildAdapterRegistry` (a controlled fixture subprocess, no real CLI hit) ‚Üí the real graph ‚Üí real G1/G2/G3 interrupt/resume ‚Üí happy path all the way to Apply" ‚Äî is actually wired together, not a pile of isolated green tests that merely give the illusion of being connected.

## 2. Goals / Non-goals

**Goals**:
- `src/loop/types.ts`: type contracts for `LoopState` (the LangGraph `Annotation.Root` state shape), `LoopNodeName`, `GateType`, `GateDecision`, `GatePayload`, `GateResumeValue`, `GateLogEntry`, etc.
- `src/loop/errors.ts`: `UnhandledGateDecisionError` (thrown when G2 receives a "proactive escalation" decision that A4a has no route for ‚Äî fail loud).
- `src/loop/workflow-def.ts`: a single naming source for node names/gate types (`LOOP_NODES`/`GATE_TYPES` constants) + a descriptive `CODER_TESTER_LOOP_DEFINITION` (A4a's landing of the DESIGN ¬ß6 "WorkflowDefinition Registry" concept ‚Äî see ¬ß5 for the detailed explanation and the rationale for scaling it down).
- `src/loop/nodes/coder.ts` / `src/loop/nodes/tester.ts`: draft/review node factory functions, reusing the real `ProviderRouter`+`PromptComposer`+`SchemaValidator` ‚Äî no new model-invocation path of any kind.
- `src/loop/gates.ts`: G1/G2/G3 gate node factories (sharing the `interrupt()` mechanism) + three routing functions (`routeAfterG1`/`routeAfterReview`/`routeAfterG2`/`routeAfterG3`).
- `src/loop/checkpoint.ts`: `createSqliteCheckpointer(dbPath)`, wired per the `SqliteSaver.fromConnString()` pattern proven in spike Q4.
- `src/loop/graph.ts`: `buildLoopGraph()` (pure graph structure, no checkpointer) + `compileLoopGraph()` (wires in a checkpointer, produces a compiled graph that can be `invoke`d/`getState`'d), connecting all the nodes/gates above into a complete state machine per DESIGN ¬ß4 (minus the Escalation subtree), including the first-ever verification of `addConditionalEdges`.
- **Hard vertical slice**: one end-to-end test proving the entire happy-path chain is wired together ‚Äî "real `MemoryStore`+`ContextInjector`+`PromptComposer` ‚Üí real `buildAdapterRegistry` (cli-bridge adapter, controlled fixture subprocess) ‚Üí real `ProviderRouter` ‚Üí real compiled graph (real `SqliteSaver` checkpointer) ‚Üí G1 interrupt+resume (approve) ‚Üí tester passes ‚Üí G3 interrupt+resume (approve) ‚Üí Apply."

**Non-goals (explicitly not doing, deferred to A4b/A5 or a later increment)**:
- ‚ùå **Threshold-escalation hard branch** (the `reject_count >= threshold` conditional route + Escalation node + human decision + Cancel terminal state) ‚Äî already settled in ¬ß0, A4b. `reject_count` itself does increment in A4a (whenever Review sends something back), but no code reads it to make a routing decision.
- ‚ùå **G2's "proactive escalation" branch actually routed** ‚Äî does A4a's `GateResumeValue.decision` type technically allow an external caller to pass a value other than `"approved"`/`"rejected"`? **No** ‚Äî A4a's `GateDecision` only has the two values `"approved" | "rejected"` (a subset of DESIGN ¬ß5's `approvals.decision` enum, minus `"override"`, which only serves escalation). The real question is: where does G2 go when it receives `"rejected"`? DESIGN's diagram shows G2 with only two outgoing edges ‚Äî "approve‚ÜíFix" and "proactively escalate‚ÜíEsc" ‚Äî no third "reject‚Üíwhere" edge. A4a makes an explicit call on this corner that DESIGN leaves undefined: **G2 only handles `"approved"`**; when it receives `"rejected"`, `routeAfterG2` throws `UnhandledGateDecisionError` rather than silently routing to some branch I invented myself. This is itself an example of "the decision wasn't fully defined, so I made a judgment call" ‚Äî flagged `[?]` for confirmation (see ¬ß9).
- ‚ùå **Persisting the three audit tables** (creating and writing to `workflow_runs`/`structured_claims`/`approvals`) ‚Äî A4b. A4a's `GateLogEntry` (¬ß4) is an **in-memory shadow** of what these three tables (especially `approvals`) will eventually hold on disk ‚Äî its field names are deliberately close to the `approvals` table's columns so A4b can consume it directly, but A4a itself **does not create any tables and does not touch any SQLite schema**.
- ‚ùå **Productionizing cross-process checkpoint resume** (storing `langgraph_thread_id` in the `workflow_runs` table, the full real-world "resume from whichever gate it was at after a restart" flow) ‚Äî A4b. A4a only delivers `checkpoint.ts` itself + one test proving "not closure-held state" (¬ß6 B4: two independently constructed compiled-graph+checkpointer instances within the same process, rather than spike Q4's approach of actually spawning two `node` child processes ‚Äî rationale in ¬ß9.1).
- ‚ùå **A colorful TUI / y/n approval interface** ‚Äî A5. A4a's G1/G2/G3 gates are driven directly by test code constructing `Command({resume: {...}})` to inject decisions; there is no human-interactable interface of any kind.
- ‚ùå **Real coder/tester model logic** ‚Äî A4a's nodes directly reuse the `ProviderRouter`+adapter+`SchemaValidator` already built in A2/A3, adding no new logic about "how to get the model to write/review code better"; the persona text (`profiles/subscription/personas/{coder,tester}.md`) carries over unchanged from A1's existing version.
- ‚ùå **Actually writing the diff to the workspace filesystem** (DESIGN ¬ß3 sequence's "write files to the workspace") ‚Äî the `Apply` node only terminates state (marks `state.coderOutput`/the G3 decision as "done," and doesn't proceed further) ‚Äî it does not call any filesystem-write API. Actually applying a unified diff to workspace files is an independent, fairly complex concern (patch-library selection, conflict handling, whether to shell out to real `git apply`), a different dimension of work from "the LangGraph orchestration mechanism itself," and is not part of this increment; the ¬ß0 table already marks this item "A4a does it, but scaled down."
- ‚ùå **Runtime dynamic loading / JSON-file-driven compilation of `WorkflowDefinition`** ‚Äî see the detailed explanation in ¬ß5's `workflow-def.ts` task description: `graph.ts`'s actual graph structure is built from type-safe, hand-written chained calls (`addNode().addEdge()...`), not runtime interpretation of a JSON file; this PRD **does not create** `workflows/coder-tester-loop.json` (the file DESIGN ¬ß6 mentions) ‚Äî see ¬ß5/¬ß9 for the rationale and impact.
- ‚ùå **Any code reading `config.yaml`'s `reject_threshold`** ‚Äî A4a's graph does no threshold evaluation; this existing config field (`profiles/subscription/config.yaml`'s `workflow.reject_threshold: 2`) remains in its current "exists but unread" state until A4b.

## 3. User stories

- As **A4a's own vertical-slice test**, I want proof that a real prompt goes from assembly, to coder producing a diff, to human approval routing it to tester, to tester passing it, to a final human sign-off, to state termination ‚Äî all the way through, entirely via the real `ProviderRouter`+adapter+`SchemaValidator`+LangGraph graph ‚Äî not through a handful of unrelated unit tests stitched together to create the illusion of "looks wired up."
- As a **future A4b developer**, I want `graph.ts`/`gates.ts` to already have G1/G2/G3's normal branches and the `reject_count` increment point ‚Äî everything unrelated to "threshold escalation" ‚Äî built out, so that A4b only needs to insert one conditional edge for the threshold check (near `routeAfterReview` or adjacent to it) + build the `Escalation` node, without having to redesign the graph's skeleton from scratch.
- As the **commander**, I want to see a test that genuinely runs a compiled graph to a G1 interruption, discards that compiled-graph object, then uses a brand-new compiled-graph+checkpointer instance (same `thread_id`, same sqlite file path) to resume it to completion ‚Äî proving that resume relies on the checkpoint on disk, not on some closure variable still sitting around in the JS process.
- As the **commander**, I want confirmation that this increment does not hit real `claude`/`codex` CLIs and burn subscription quota in automated tests ‚Äî the test strategy carries forward A3's already-established "real cli-bridge adapter spawn, but targeting a controlled fixture script" pattern.

## 4. Data model

This increment **creates no tables and touches no SQLite** ‚Äî the three audit tables `workflow_runs`/`structured_claims`/`approvals` remain A4b's job (continuing the "stateless" boundary set in A2/A3 ¬ß4). The only "data shape" here is LangGraph's in-memory state + checkpoint state:

```typescript
// src/loop/types.ts ‚Äî suggested shape; field names may be fine-tuned during the build phase, but each field's semantics/existence below is a PRD hard constraint
const LoopState = Annotation.Root({
  task: Annotation<string>(),                              // the original task description, unchanged for the entire run
  feedback: Annotation<string | undefined>(),               // the feedback the next draft node needs to see (from G1's rejection reason / the tester issues G2 carries along on approval / G3's rejection reason); consumed and cleared by the draft node, does not accumulate indefinitely
  injectedContext: Annotation<ContextInjectionResult>(),    // the result of ContextInjector.inject(), injected once by the caller outside the graph (see ¬ß5, "why ContextInjector isn't called again inside a node"), unchanged for the entire run
  coderOutput: Annotation<CoderOutput | undefined>(),       // output of the most recent draft node run
  coderResult: Annotation<InvokeResult | undefined>(),      // same, the raw InvokeResult (includes provider/model/toolExecChecked)
  testerOutput: Annotation<TesterOutput | undefined>(),     // output of the most recent review node run
  testerResult: Annotation<InvokeResult | undefined>(),
  rejectCount: Annotation<number>({ reducer: (_a, b) => b, default: () => 0 }), // increments whenever Review sends something back; A4a has no code reading it to make routing decisions (¬ß0/¬ß2 Non-goals) ‚Äî purely, the "Inc" step is built out first
  g1Decision: Annotation<GateDecision | undefined>(),
  g2Decision: Annotation<GateDecision | undefined>(),
  g3Decision: Annotation<GateDecision | undefined>(),
  gateLog: Annotation<GateLogEntry[]>({ reducer: (a, b) => a.concat(b), default: () => [] }), // see explanation below
  applied: Annotation<boolean>({ reducer: (_a, b) => b, default: () => false }),
});

interface GateLogEntry {
  gate: GateType;            // "G1_SEND_TO_TESTER" | "G2_SEND_TO_FIX" | "G3_FINAL_MERGE" (the A4a subset of DESIGN ¬ß5's approvals.gate_type enum, excluding "ESCALATION_ACK" which only exists in A4b)
  decision: GateDecision;    // "approved" | "rejected"
  reasoningText?: string;    // the reasoning text a human provides at resume time
  decidedAt: string;         // ISO timestamp
}
```

**`gateLog` is a deliberately designed forerunner to A4b**: its field names are close to DESIGN ¬ß5's `approvals` table columns (`gate_type`/`decision`/`reasoning_text`/`decided_at`), so that when A4b adds audit persistence, it can just write each entry of `state.gateLog` into the `approvals` table ‚Äî no need to redesign this layer of "what information needs to be remembered during graph execution." **Accurate phrasing (Zorro R1 M1 correction)**: `gateLog` is an Annotation channel of `LoopState`; when the real graph is compiled with `SqliteSaver`, LangGraph serializes the **entire state (including gateLog)** into its own checkpoint tables (`checkpoints`/`writes`) ‚Äî it is not "in-memory only, gone the instant the process exits." The real boundary between A4a and A4b is: A4a never writes `gateLog` entries into the business audit table `approvals` that A4b builds; LangGraph's own checkpoint persistence and aeloop's own `approvals` persistence are two independent things, and only the latter is A4b's job. This is a deliberate design choice, not an oversight ‚Äî flagging it for confirmation (¬ß9).

LangGraph's own checkpoint tables (`checkpoints`/`writes`, auto-created by `SqliteSaver.setup()`) are a completely different thing from the aeloop business state above ‚Äî spike-findings.md ¬ß4 has already proven these two sets of tables are entirely independent: the former stores "which step the graph is on, what the state snapshot is" (LangGraph manages this itself, A4a uses it directly), the latter is aeloop's own business-audit semantics (built only in A4b).

  - `LOOP_NODES`/`GATE_TYPES` move to `workflow-def.ts` (below); `types.ts` only imports their types and does not redefine the constants.

- `src/loop/errors.ts` (**new file**):
  - `UnhandledGateDecisionError extends Error`: constructor `(gate: GateType, decision: string)`, with a message explaining "A4a has no routing target built for this gate/decision combination, wait for A4b." `routeAfterG2` throws this when it receives a decision other than `"approved"` (¬ß2 Non-goals already explains why).

### `workflow-def.ts` (A4a's landing of the WorkflowDefinition concept, scaled down ‚Äî please double-check this scaling-down)

- `src/loop/workflow-def.ts` (**new file**):
  - `LOOP_NODES` (an `as const` object, the six real nodes: `draft`/`g1`/`review`/`g2`/`g3`/`apply`) + `GATE_TYPES` (`as const`, the three gate types) ‚Äî this is the **single source of truth** for every node-name/gate-type string literal used in `graph.ts`/`gates.ts`/`nodes/*.ts`, avoiding the way the spike scripts hand-wrote the same string in multiple places.
  - `CODER_TESTER_LOOP_DEFINITION`: a **descriptive** (not executable) data structure listing `LOOP_NODES`'s node set + every edge (including which ones are conditional), roughly shaped like:
    ```typescript
    export const CODER_TESTER_LOOP_DEFINITION = {
      id: "coder-tester-loop",
      nodes: Object.values(LOOP_NODES),
      edges: [
        { from: "__start__", to: LOOP_NODES.draft },
        { from: LOOP_NODES.draft, to: LOOP_NODES.g1 },
        { from: LOOP_NODES.g1, to: [LOOP_NODES.review, LOOP_NODES.draft], conditional: true },
        { from: LOOP_NODES.review, to: [LOOP_NODES.g3, LOOP_NODES.g2], conditional: true },
        { from: LOOP_NODES.g2, to: [LOOP_NODES.draft], conditional: true },
        { from: LOOP_NODES.g3, to: ["apply", LOOP_NODES.draft], conditional: true },
        { from: LOOP_NODES.apply, to: "__end__" },
      ],
    } as const;
    ```
  - **This is a scaling-down of DESIGN ¬ß6's "graph.ts is compiled from a WorkflowDefinition" design, and needs the strategist/commander to sign off on**: spike-findings.md suggestion #2 explicitly points out that "dynamically generating a StateGraph from JSON/YAML" was never verified for feasibility, and LangGraph's TS API (as seen in spike Q2/Q3/Q4, all three scripts) uses **hand-written chained calls** (`new StateGraph(State).addNode("draft", draftNode).addEdge(...)`) under a type-safe regime, not "loop over an array reading config and calling `addNode`" as some kind of runtime dynamic structure ‚Äî a graph's node/edge **types** (used for `Annotation`'s state-shape inference, `Command`'s `Nodes` generic) are inherently compile-time literals in LangGraph's TS type system; forcibly generating them at runtime from a data structure would either lose this layer of type safety (falling back to `any`/`string`) or require fairly elaborate TS type gymnastics (well beyond what a single PRD batch should attempt). A4a's choice is: **`CODER_TESTER_LOOP_DEFINITION` is "human-readable documentation + a single naming source," not "config that `graph.ts` reads and interprets at runtime"** ‚Äî `graph.ts`'s actual `addNode`/`addEdge`/`addConditionalEdges` calls are hand-written, but every string literal they use is imported from `LOOP_NODES`/`GATE_TYPES`, so there's no risk of "one copy in the docs, another in the code, and the two can drift apart" (TS will error at compile time on an import of a nonexistent property). This PRD **does not create** `workflows/coder-tester-loop.json` (the file DESIGN ¬ß6 mentions) ‚Äî a JSON file that "looks like it's being read but actually isn't" is more misleading than not having the file at all; once a second real workflow that genuinely needs dynamic loading shows up (DESIGN ¬ß1.7 itself says "harden this once there are 2-3 real workflow needs"), it can be decided then whether to turn `CODER_TESTER_LOOP_DEFINITION` into a JSON file actually read at runtime. **If this call doesn't match what the commander originally meant by DESIGN ¬ß6's sentence, please just change it ‚Äî I won't quietly implement it my own way in later work.**

### `nodes/coder.ts` / `nodes/tester.ts`

- `src/loop/nodes/coder.ts` (**new file**):
  - `createDraftNode(deps: { router: ProviderRouter; composer: PromptComposer }): (state: LoopStateType) => Promise<Partial<LoopStateType>>`.
  - Node body: ‚ëÝ assemble the task text to use this round using `state.feedback` (may be `undefined`) ‚Äî when `state.feedback` exists, `task = state.task + "\n\n---\n\nFeedback from the previous round:\n" + state.feedback`; when it doesn't, it's just the original `state.task` (**this reuses the "append, don't replace" convention already established by `SchemaValidator.buildRetryPrompt()`**, not a new pattern I invented ‚Äî consistent with the rest of this codebase); ‚ë° `deps.composer.compose("coder", state.injectedContext, task)` to get the prompt; ‚ë¢ `deps.router.route("coder")` to get the adapter; ‚ë£ `new SchemaValidator().validate({ schema: CoderOutput, request: { role: "coder", prompt }, invoke: (req) => adapter.invoke(req) })` (`SchemaValidator` has no constructor dependencies, just `new` it directly, consistent with how it's used in `harness-cli.e2e.test.ts` ‚Äî no extra injection needed); ‚ë§ return `{ coderOutput: data, coderResult: result, feedback: undefined }` (**clear feedback once it's consumed**, to prevent it from being appended twice in a row if this node is somehow re-entered before some gate has updated `state.feedback` again).
  - `SchemaValidationError` (an existing A2 type) is **not caught separately here** ‚Äî if the node function throws, it lets `compiled.invoke()` throw as a whole, and that's expected behavior (coder producing non-compliant output twice in a row is a failure the caller needs to know about, not something the Loop layer should silently swallow).
- `src/loop/nodes/tester.ts` (**new file**):
  - `createReviewNode(deps: { router: ProviderRouter; composer: PromptComposer }): (state: LoopStateType) => Promise<Partial<LoopStateType>>`.
  - Node body: ‚ëÝ a defensive check that `state.coderOutput` exists (in theory the graph's edges guarantee the review node is only ever invoked after draft, but a node function shouldn't trust the implicit assumption that "the graph structure guarantees this" ‚Äî throw a clear `Error` when it's missing, not an `undefined.diff`-style bare crash); ‚ë° assemble the task text: feed the original task + coder's diff + coder's claims all to tester (a literal application of DESIGN ¬ß3's sequence "invoke(tester, diff+ctx, schema)" ‚Äî claims are included too because the tester persona's house rules explicitly state "Verify claims instead of just reading and agreeing with them" ‚Äî without a claims list, tester has nothing specific to verify against); ‚ë¢ call `composer`/`router`/`SchemaValidator` the same way as above, `schema: TesterOutput`; ‚ë£ return `{ testerOutput: data, testerResult: result, rejectCount: data.verdict === "reject" ? state.rejectCount + 1 : state.rejectCount }` (DESIGN ¬ß4's "Inc" step, happening unconditionally here, not dependent on any threshold check ‚Äî ¬ß0 already explains this belongs in A4a).

### `gates.ts` (G1/G2/G3)

- `src/loop/gates.ts` (**new file**):
  - A shared internal factory `createGateNode(gate: GateType, buildPayload: (state) => Omit<GatePayload, "gate">, deriveFeedback: (state, resume: GateResumeValue) => string | undefined)`, returning a node function whose body:
    1. `const payload: GatePayload = { gate, ...buildPayload(state) }` (a pure function, reading state without side effects ‚Äî **this step reruns on resume**, the direct application of spike-findings.md Q3's "one behavior that isn't a bug but affects how `nodes/` code is written" ‚Äî so this step must never have a "send a notification once"/"write a log entry once" kind of non-idempotent side effect).
    2. `const resume = interrupt(payload) as GateResumeValue` (the first call genuinely pauses here; on resume this line directly returns the value from the external `Command({resume})` call, without pausing again ‚Äî the behavior already verified by spike Q3).
    3. Only **after** `interrupt()` is any side-effecting operation allowed: constructing the `GateLogEntry` (`decidedAt: new Date().toISOString()`).
    4. Return that gate's own decision field (`g1Decision`/`g2Decision`/`g3Decision`, specified individually by each of the three concrete gates that call `createGateNode`, not guessed by this shared factory itself) + `feedback: deriveFeedback(state, resume)` + `gateLog: [entry]` (the `reducer` is `concat`, accumulating rather than overwriting).
  - The three concrete gates:
    - `createG1Node()` = `createGateNode("G1_SEND_TO_TESTER", (state) => ({ diffRef: state.coderOutput?.diff, question: "approve sending this diff to the tester?" }), (_state, resume) => resume.reasoningText)` (when G1 rejects, the feedback is simply whatever reason the human gave on the spot ‚Äî there's no further context to add).
    - `createG2Node()` = `createGateNode("G2_SEND_TO_FIX", (state) => ({ issues: state.testerOutput?.issues, question: "approve sending the tester's findings back to the coder for a fix?" }), (state, resume) => [state.testerOutput?.issues?.join("; "), resume.reasoningText].filter((s): s is string => Boolean(s)).join("\n\n"))` (once G2 approves, what coder actually needs to see **is primarily tester's issue list** ‚Äî the human's `reasoningText` supplements it, it doesn't replace it ‚Äî if this only used `resume.reasoningText`, coder would see nothing of what tester actually said, meaning the gate approved but the feedback never made it through ‚Äî a real functional gap, not over-engineering on my part).
    - `createG3Node()` = `createGateNode("G3_FINAL_MERGE", (state) => ({ diffRef: state.coderOutput?.diff, question: "final sign-off: apply this diff?" }), (_state, resume) => resume.reasoningText)`.
  - The three routing functions (for `graph.ts`'s `addConditionalEdges` to use):
    - `routeAfterG1(state): "review" | "draft"` ‚Äî `"approved"` ‚Üí `"review"`; `"rejected"` ‚Üí `"draft"`; other values are in theory already excluded by the type system, but there's still a runtime `default: throw` fallback (not silent).
    - `routeAfterReview(state): "g3" | "g2"` ‚Äî `state.testerOutput.verdict === "pass"` ‚Üí `"g3"`; `"reject"` ‚Üí `"g2"`.
    - `routeAfterG2(state): "draft"` ‚Äî `state.g2Decision === "approved"` ‚Üí `"draft"`; anything else (including `"rejected"` ‚Äî ¬ß2 Non-goals already explains why A4a deliberately gives it no routing target) ‚Üí throw `UnhandledGateDecisionError("G2_SEND_TO_FIX", state.g2Decision ?? "undefined")`.
    - `routeAfterG3(state): "apply" | "draft"` ‚Äî `"approved"` ‚Üí `"apply"`; `"rejected"` ‚Üí `"draft"`.

### `checkpoint.ts`

- `src/loop/checkpoint.ts` (**new file**):
  - `createSqliteCheckpointer(dbPath: string): SqliteSaver` ‚Äî just `SqliteSaver.fromConnString(dbPath)` (the usage spike Q4 proved out); the only reason for this thin wrapper is so `graph.ts`/test code all get a checkpointer from this one function, instead of each separately importing `@langchain/langgraph-checkpoint-sqlite`.
  - **Not doing**: an equivalent wrapper for `MemorySaver` ‚Äî A4a's graph always defaults to a `SqliteSaver` (even in test scenarios, pointing at a temp file), with no option for "pure in-memory, discarded on process exit," because "can it be resumed from outside" is part of the entire point of the G1/G2/G3 gates existing (explicitly required by the ¬ß8 acceptance criteria) ‚Äî offering a default state-losing option would be too easy for future developers to misuse.

### `graph.ts`

- `src/loop/graph.ts` (**new file**):
  - `buildLoopGraph(deps: { router: ProviderRouter; composer: PromptComposer }): StateGraph<...>` (the uncompiled graph, pure structure):
    - `addNode(LOOP_NODES.draft, createDraftNode(deps))`
    - `addNode(LOOP_NODES.g1, createG1Node())`
    - `addNode(LOOP_NODES.review, createReviewNode(deps))`
    - `addNode(LOOP_NODES.g2, createG2Node())`
    - `addNode(LOOP_NODES.g3, createG3Node())`
    - `addNode(LOOP_NODES.apply, applyNode)` (see below ‚Äî a small unexported function internal to graph.ts, not worth its own `nodes/apply.ts` file ‚Äî DESIGN ¬ß6's file list doesn't list one either, consistent with ¬ß2 Non-goals' "Apply only terminates state," and it's simple enough that a dedicated file isn't warranted)
    - `addEdge(START, LOOP_NODES.draft)`
    - `addEdge(LOOP_NODES.draft, LOOP_NODES.g1)`
    - `addConditionalEdges(LOOP_NODES.g1, routeAfterG1)`
    - `addEdge(LOOP_NODES.g1, LOOP_NODES.review)` is **not** meant to be a plain edge ‚Äî since `addConditionalEdges` above already covers the target set `{review, draft}`, don't add a redundant `addEdge` here (an easy mistake to make while first writing the code ‚Äî the PRD flags it in advance).
    - `addConditionalEdges(LOOP_NODES.review, routeAfterReview)`
    - `addConditionalEdges(LOOP_NODES.g2, routeAfterG2)`
    - `addConditionalEdges(LOOP_NODES.g3, routeAfterG3)`
    - `addEdge(LOOP_NODES.apply, END)`
  - `applyNode(state): Partial<LoopStateType>` ‚Äî an internal function, `return { applied: true }`, no filesystem reads/writes (¬ß2 Non-goals).
  - `compileLoopGraph(graph, checkpointer): CompiledGraph` ‚Äî `graph.compile({ checkpointer })`, a thin wrapper whose only reason for existing is so every call site (tests + the future A5) goes through this one function, instead of each separately repeating `.compile({...})`.
  - **A hard requirement on implementation order (echoing spike suggestion #3)**: `addConditionalEdges` is the one LangGraph mechanism in this entire PRD that the spike completely never verified (all 5 of the spike's Qs are either linear graphs or a single `interrupt`, none used conditional edges). ¬ß6's batch breakdown puts "use toy nodes first to verify `addConditionalEdges` can correctly route based on return value" as the very earliest batch (the start of B3), rather than leaving it to be discovered as a problem right at the end.

## 9. Dependencies / Risks / Open questions

### 9.1 Proving checkpoint's "not closure-held state": same-process two-phase vs. spike Q4's real cross-process ‚Äî recommend adopting same-process two-phase

The dispatch instructions' original wording was "doesn't have to be cross-process, but must prove resume isn't relying on closure-held state" ‚Äî ¬ß5/¬ß6 of this PRD choose the "within a single test file, construct two structurally independent compiled-graph+checkpointer instances, explicitly discard the first once it's used" same-process approach, rather than literally copying spike Q4's approach of using `child_process` to spawn two independent `node` child processes. **Rationale**: spike Q4 already proved "genuine cross-process" once, during issue #13's pre-issue stage (pid 27363‚Üípid 27564, zero shared memory) ‚Äî A4a doesn't need to re-prove "the LangGraph checkpoint mechanism itself is trustworthy," a fact that already has evidence, in every single batch; what the A4a test suite actually needs to guard against is "did aeloop's own `checkpoint.ts`/`graph.ts` accidentally introduce some kind of in-process singleton/cache that makes resume look like it works while secretly depending on some variable left over from the first `invoke()`" ‚Äî same-process two-phase (explicitly discarding the reference + constructing a brand-new instance) catches this exact problem just as well, while actually spawning child processes would introduce extra test-infrastructure complexity (subprocess orchestration, cross-process log collection, CI environment subprocess permissions) for the benefit of re-verifying something already verified. **If the commander believes "a fresh external call" must literally mean a new `node` process to count, please just say so and I'll redo this test following spike Q4's pattern.**
