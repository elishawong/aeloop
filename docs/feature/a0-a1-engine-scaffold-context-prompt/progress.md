### B4 — context/confirmation.ts (ConfirmationService) + confirmation.test.ts
- Status: complete
- commit: `b24fa3f` — `feat(context): B4 — ConfirmationService (confirm/correct/reject, transactional)`
- What was done:
  - `confirmation.ts`: `ConfirmationService.confirm(memoryId, actor, now?)` / `.correct(memoryId, newContent, actor, now?)` / `.reject(memoryId, actor, now?)`, all three wrapped in `store.runInTransaction(() => {...})`, covering both the memories write and the `memory_confirmations` insert.
  - **`correct()` semantics (self-implemented per PRD §9.0#5, without reading the Verity internal-network source)**: `old_content` is always taken as the memory's `content` "at the moment right before this call" (not the original insert value), so consecutive `correct()` calls correctly chain ("latest" rather than "original"). `correct()` also marks `confidence_state` as `confirmed` (correcting the content is itself a confirmation of the new content).
  - **`reject()` semantics**: sets `confidence_state = 'rejected'`, but **deliberately does not clear** `confirmed_at`/`confirmed_by` — these two columns record "when/by whom it was last confirmed," and rejecting shouldn't erase that piece of history; the complete action history always lives in `memory_confirmations` (the reject's own row has its own independent `actor`/timestamp). This design decision is documented in the class's docstring.
- **Required-item assertion points** (mapped item-by-item to which `it` in the test file):
  - **Transaction atomicity** (`describe("ConfirmationService — transaction atomicity ...")`, 3 `it`s): `vi.spyOn(store, "insertConfirmation").mockImplementation(() => { throw ... })`, injecting the failure right after the memories write has **already executed** inside each of `confirm()`/`correct()`/`reject()`; assertion points — for `confirm`: after rollback, `getMemoryById().confidenceState === "unconfirmed"` and `confirmedAt`/`confirmedBy` are still `null`; for `correct`: after rollback, `content` is still `"original"` (not changed to the new content) and `confidenceState` is still `"unconfirmed"`; for `reject`: after rollback, `confidenceState` is still `"unconfirmed"` (not mistakenly marked `"rejected"`). All three additionally assert that `store.getConfirmationsForMemory(id)` is an empty array (no half-written audit row left behind).
  - **No existing confirmation record path** (`correct()`, 2 `it`s): the first calls `correct()` directly on a brand-new memory that has **never had confirm/correct/reject called on it**, asserting `getConfirmationsForMemory` is `[]` before the call and that after the call `old_content` equals the original inserted content; the second immediately calls `correct()` a second time, asserting the second confirmation row's `old_content` equals the **content after the first correct**, not the original content — proving the "latest" semantics really works and isn't a coincidence.
- Local self-check: `pnpm build`/`pnpm lint`/`pnpm test` all green, **52/52 passed** (11 new).
